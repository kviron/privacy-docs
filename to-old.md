# Введение {#введение}

“Wandwork”- web-приложение, которое помогает заказчикам оперативно формировать команды для проектов, а фрилансерам \- находить подходящую работу.

В рамках проекта будет разработано web-приложение для фрилансеров и заказчиков, а также создана панель администрирования web-приложения.

# Функциональные требования {#функциональные-требования}

## Вход в web-приложение {#вход-в-web-приложение}

Использование web-приложения доступно только авторизованным пользователям. 

Перед регистрацией и авторизацией система должна предоставлять возможность ознакомиться с текстом политики обработки персональных данных.  

Регистрация и авторизация в web-приложении осуществляется по e-mail.

**Регистрация**

Для регистрации в web-приложении необходимо:

- ввести e-mail адрес;  
- подтвердить e-mail с помощью кода, отправленного на e-mail.

Код для проверки действует 10 минут.

Если пользователь пытается ввести неактуальный код или код, который не совпадает с отправленным системой, приложение должно отображать ошибку. Если пользователь некорректно ввел код 3 раза система блокирует возможность отправки кода на 1 час. 

Система позволяет повторно отправить код спустя 60 секунд после отправки запроса на получение кода.

После повторной отправки кода предыдущий код становится недействительным.

Если пользователь запрашивает код больше 3 раз в течение 1 часа с первого запроса \- система блокирует возможность запроса кода на 1 час. Счетчик блокировки сбрасывается после успешной активации кода.

Данные настройки можно изменить в административной панели.

У пользователя появляется возможность задать пароль учетной записи. 

Пароль должен соответствовать следующим правилам:

- длина пароля \- не менее 8-ми символов и не больше 24-х;  
- пароль должен содержать символы латиницы;  
- пароль должен содержать специальные символы;  
- пароль не может совпадать с логином.

Заданный пароль нужно подтвердить, введя его еще раз.

После подтверждения пароля система создает аккаунт пользователя и авторизует его.

**Авторизация**

Для авторизации по e-mail пользователь должен указать e-mail и пароль.

E-mail и пароль должны совпадать с сохраненными в системе. В противном случае система уведомляет пользователя об ошибке, а авторизация невозможна. Если e-mail и пароль совпали с сохраненными в системе, система авторизует пользователя.

**Восстановление пароля**

Система должна предоставлять возможность восстановления пароля. 

Для восстановления пароля пользователь должен ввести e-mail. Если e-mail зарегистрирован, система отправляет на него код.

Система должна предоставлять возможность повторной отправки кода. Условия проверки кода и повторного запроса совпадают с описанными в разделе “Регистрация”.

Пользователь должен ввести полученный код для входа. Если код введен неверно, система уведомляет пользователя об ошибке.

Если код введен верно, система предоставляет пользователю возможность ввести новый пароль. 

Новый пароль должен быть введен дважды. Если пароли совпали система меняет пароль.

После успешного восстановления пароля система прерывает все сеансы пользователя и предлагает авторизоваться пользователю с использованием нового пароля.

## Выход из web-приложения {#выход-из-web-приложения}

Авторизованный пользователь должен иметь возможность выйти из аккаунта web-приложения. 

## Профиль пользователя {#профиль-пользователя}

После регистрации пользователю становится доступно заполнение профиля.

Для работы с профилем доступны следующие возможности:

- заполнение профиля;  
- редактирование профиля.

### Заполнение профиля {#заполнение-профиля}

После прохождения регистрации пользователю доступен выбор роли:

- заказчик;  
- специалист;

и заполнение общих данных профиля: 

| Данные | Обязательно / необязательно | Комментарий |
| :---- | :---- | :---- |
| Имя | обязательно | строка |
| Фамилия | обязательно | строка |
| Фотография | необязательно | Файл с изображением.Требования к файлу: расширение файла: jpg, png; |

После заполнения основного профиля пользователю становятся доступны возможности:									

- работа с пользователями (подробное описание инструмента в разделе “[Круги мэйтов](#круги-мэйтов)”);  
- работа с сообщениями (подробное описание инструмента в разделе “[Чат](#чат)”);  
- заполнение профиля “Специалист”;  
- заполнение профиля “Заказчик”;  
- создание проектов для роли Заказчик (подробное описание в разделе “[Создание проектов](#создание-проектов)”).

Пользователь может переключаться между ролями.

Для профиля “Заказчик” доступны возможности:

- управление проектами;  
- управление специалистами на проектах.

Для профиля “Специалист” доступны возможности:

- отклики на проект;  
- участие в проекте.

Пользователь может пропустить заполнение основных данных.

**Профиль заказчика**

При заполнении профиля заказчика пользователь должен дополнительно указать информацию:

### Пустой профиль {#пустой-профиль}

Пустым считается профиль, в котором не заполнены:

- общие данные профиля (имя, фамилия).

Пользователь с пустым профилем может:

- просматривать проекты \- под ролью специалист;  
- добавлять вакансии в избранное \- под ролью специалист;  
- просматривать мэйты.

Пользователь с пустым профилем не может:

- пользоваться функционалом чата;  
- отображаться в мэйтах;  
- рекомендовать пользователей и быть рекомендованным;  
- откликаться на вакансии \- под ролью специалист;  
- отображаться в подборке специалистов на вакансию \- под ролью специалист;  
- быть приглашенным на вакансию \- под ролью специалист;  
- создавать проекты \- под ролью заказчик.

Если пользователь зарегистрировался по ссылке-приглашению, то в первый круг мэйтов он попадет только после заполнения имени и фамилии.

Специалист с заполненными основным данными, но неопубликованным профилем специалиста:

- может пользоваться функционалом чата;  
- может рекомендовать пользователей и быть рекомендованным;  
- может отображаться в мэйтах;  
- не может отображаться в подборке специалистов на вакансию;  
- не может быть приглашенным на вакансию;  
- не может откликаться на вакансии.

### Редактирование профиля {#редактирование-профиля}

Пользователям доступно редактирование любых указанных им данных в профиле.

**Редактирование e-mail**

Система должна предоставлять пользователю возможность редактировать e-mail. Новый e-mail должен отличаться от предыдущего.

После ввода e-mail система должна проводить проверку e-mail, с помощью кода, аналогичную проверке при регистрации (подробное описание в разделе “Регистрация”).

Если новый e-mail проходит проверку \- система сохраняет его. После \- система прекращает все сеансы для аккаунта и предлагает пользователю авторизоваться с новым e-mail.

**Смена пароля**

У пользователя должна быть возможность смены пароля. 

Смена пароля возможна только после проверки действующего пароля. Если пароль введен неправильно, система информирует об ошибке.

После ввода нового пароля система должна проверять его, запросив у пользователя ввести пароль снова. Если пароли совпадают \- система меняет пароль.

Пароль должен отличаться от предыдущего.

Требования к паролю описаны в разделе “Регистрация”.

После смены пароля система должна прервать все активные сеансы пользователя. Пользователь должен повторно авторизоваться с новым паролем.

### Удаление профиля

У пользователя должна быть возможность удалить профиль.

Перед удалением профиля система должна запрашивать подтверждение действия.

Удаленный профиль восстановить нельзя.

После удаления профиля система должна удалять данные для входа пользователя. Заново воспользоваться этими данными пользователь может только после регистрации.

Личные данные пользователя удаляются.

Все данные профиля специалиста \- удаляются.

Все данные профиля заказчика \- удаляются.

Если у пользователя было две роли, то удаляются данные обеих ролей.

Если у пользователя есть проекты, отличные от статуса “Завершен” \- они переводятся в статус “Завершен”. Реакции системы на событие “Профиль заказчика удален” перечислены в разделе “[События](https://docs.google.com/document/d/1PKfooimu-J7Ov1wR3e649pxQ_5f6Pr7jmQORcI4X23E/edit?tab=t.0#heading=h.txlo2x7ranp9)”.

Если у пользователя есть активные приглашения, или отклики \- они удалятся навсегда.

Если пользователь является участником проектов \- он удаляется с вакансии. Реакции системы на событие “Профиль специалиста удален” описаны в разделе “[События](https://docs.google.com/document/d/1PKfooimu-J7Ov1wR3e649pxQ_5f6Pr7jmQORcI4X23E/edit?tab=t.0#heading=h.txlo2x7ranp9)”.

Пользователь удаляется со всех кругов мэйтов, в которых состоял.

Если у пользователя были рекомендации, пользователь удаляется из списков порекомендовавших пользователей.

Профиль удаленного пользователя становится недоступным для поиска и просмотра.

Личные чаты с удаленным пользователем остаются доступными только для просмотра. Сообщения из чатов проектов, в которых состоял пользователь остаются. Фамилия и имя удаленного пользователя меняются на фразу, обозначающую удаление пользователя.

### Просмотр профиля {#просмотр-профиля}

Система должна предоставлять возможность просмотра своего профиля

- изображение пользователя;  
- статус (Ищу проекты/Не ищу проекты \- для специалиста);  
- имя, фамилия.

Система должна предоставлять возможность просматривать информацию о профиле специалиста, если он заполнен:

- ссылки на портфолио;  
- файлы портфолио;  
- специализации:  
  - название специализации;  
  - уровень;  
- навыки;  
- индустрии;  
- языки (порядок сортировки описан ниже):  
  - название языка;  
  - уровень владения;  
- текст резюме исполнителя \- при наличии;

Порядок сортировки языков: если указан родной язык, то он отображается первым. Остальные выводятся в порядке названий, заданном в справочнике.

## Приглашение пользователей в сервис {#приглашение-пользователей-в-сервис}

В системе должен быть предусмотрен инструмент для приглашения пользователей, незарегистрированных в системе, по ссылке-приглашению.

Система должна предоставлять возможность генерировать многоразовую ссылку-приглашение для регистрации на платформе.

У пользователя должна быть возможность скопировать ссылку, чтобы отправить ее приглашенному вручную вне системы.

**Регистрация по ссылке-приглашению**

При переходе по ссылке система должна предоставлять пользователю информацию о пригласившем:

- фотография;  
- фамилия;  
- имя.

После просмотра информации о пригласившем, у пользователя должна быть возможность перейти к регистрации в web-приложении по e-mail.

Если пользователь зарегистрировался по ссылке-приглашению, он добавляется в первый круг мэйтов пригласившего пользователя после заполнения фамилии и имени.

## Мэйты {#мэйты}

### Просмотр профилей других пользователей {#просмотр-профилей-других-пользователей}

Система должна предоставлять возможность просмотра профилей других пользователей. Информация совпадает с информацией при просмотре своего профиля.

Дополнительно о других пользователях доступна информация:

- круг, в котором состоит пользователь, относительно просматривающего (подробное описание в разделе “[Круги мэйтов](#круги-мэйтов)”);  
- рекомендующие пользователи.

О рекомендовавших пользователях отображается информация:

- общее количество рекомендовавших;  
- количество общих рекомендовавших из общего количества;  
- изображение профилей до 4 (включительно) рекомендовавших;  
- оставшееся количество рекомендовавших, если их больше 4\. 

Пользователь может перейти к просмотру списка всех рекомендовавших пользователей. В списке отображаются данные пользователей:

- имя;  
- фамилия;  
- фотография;  
- круг мэйта относительно просматривающего пользователя.

Пользователи отображаются с сортировкой по умолчанию: 

- круг мэйта относительно просматривающего пользователя от 1-го до 3-го;  
- по дате создания рекомендации внутри каждого мэйта от более поздних к более ранним.

При просмотре рекомендовавших своего первого круга мэйтов пользователь выводится сам себе в списке рекомендовавших с правилом низкого приоритета. Вместо круга мэйта пользователю у своего профиля отображается текст: “Это я”. Пользователь учитывается в количестве общих рекомендовавших. В общие рекомендовавшие входят пользователи 2-го круга относительно просматривающего пользователя.

Система должна предоставлять следующие возможности:

- рекомендовать пользователя;  
- написать пользователю.

Дополнительно система предоставляет информацию:

- роль пользователя с возможностью переключаться между ними, если у пользователя две роли;  
- время последнего посещения пользователя в формате:  
  - "онлайн";  
  - "был сегодня";  
  - "был вчера";  
  - "был на этой неделе";  
  - "был более недели назад".

**Роли пользователя**

Система должна предоставлять информацию о том, в каких ролях выступает пользователь в web-приложении.

Система должна предоставлять возможность просматривать профиль каждой из ролей пользователя. 

## Время посещения пользователя {#время-посещения-пользователя}

Система должна определять, находится ли пользователь “Онлайн”. Не “Онлайн” пользователь считается, если он не проявлял активности 15 минут. 

Система должна предоставлять информацию о времени последнего посещения пользователем с помощью следующих статусов:

- "онлайн";  
- "был сегодня";  
- "был вчера";  
- "был на этой неделе";  
- "был более недели назад".

### Просмотр списка пользователей {#просмотр-списка-пользователей}

Система должна предоставлять возможность просмотра списка пользователей. 

Пользователи в списке отсортированы по кругу мэйта, в котором состоит пользователь, относительно просматривающего. В первую очередь \- 1-ый круг, далее \- 2-ой, далее \- 3-ий.

О каждом пользователе в списке предоставляется следующая информация:

- фотография пользователя \- уменьшенное изображение из профиля;  
- имя пользователя;  
- фамилия пользователя;  
- круг мэйта, относительно пользователя;  
- одна из специализаций пользователя (при наличии) \- выводится последняя из указанных пользователем специализаций;   
- количество других специализаций пользователя (при наличии);  
- файлы портфолио (при наличии).

Над каждым пользователем доступны следующие действия:

- написать пользователю \- переход к чату с пользователем;  
- рекомендовать пользователя \- для мэйтов 2-ого и 3-его круга;  
- просмотреть подробную информацию о пользователе.

Кроме того, каждый пользователь должен быть отмечен, как:

- заказчик \- при заполненном профиле заказчика;  
- фрилансер \- при заполненном профиле специалиста.

Для пользователей второго круга мэйтов должен быть указан рекомендовавший пользователь из первого круга и оставшееся количество рекомендующих пользователей.

Пользователь не видит свой профиль в общем списке пользователей.  
Если у пользователя две роли, то в списке он отображается в роли фрилансера.

**Поиск пользователей**

Система должна предоставлять возможность поиска пользователей по имени и фамилии.

**Фильтр пользователей**

Система должна предоставлять возможность фильтра пользователей. Фильтр пользователей возможен по следующим полям:

- круг мэйта;  
- специализация;  
- уровень;  
- индустрия;  
- навыки.

**Сортировка пользователей**

Система должна предоставлять возможность сортировки пользователей по дате регистрации с учетом круга мэйта:

- по возрастанию \- “сначала новые”;  
- по убыванию \- “сначала старые”.


### Круги мэйтов {#круги-мэйтов}

В web-приложении предусмотрен механизм рекомендаций пользователей \- “Круги мэйтов”.

Система должна предоставлять возможность добавить пользователя в 1-ый круг мэйтов \- рекомендовать пользователя. Реакция системы на событие “Вас порекомендовали” описана в разделе “[События](#события)”.

В системе предусмотрены следующие круги мэйтов:

- мой круг (1-й круг) \- пользователи, рекомендованные непосредственно пользователем;  
- 2-ой круг \- пользователи, рекомендованные мэйтами 1-го круга пользователя;  
- 3-ий круг \- все остальные пользователи.

Система предусматривает следующие возможности для работы с пользователями моего (1-го) круга мэйтов:

- удаление пользователя из мэйтов 1-ого круга;  
- просмотр мэйтов пользователя.


## Управление проектами {#управление-проектами}

Для размещения и управления проектами в web-приложении предусмотрен инструмент “Проекты”. Инструмент предусматривает функции как для заказчика, так и для специалиста.

Проект может находиться в следующих статусах:

| Статус | Описание | Доступные действия с проектом |  |
| :---- | :---- | :---- | :---- |
|  |  | **Заказчик (создатель)** | **Специалист** |
| Черновик | Созданный, неопубликованный проект  | просмотреть опубликовать редактировать: данные проекта создать вакансию редактировать вакансию удалить | Не видит проект |
| Опубликован | Проект после публикации с открытыми вакансиями | просмотреть начать (перевести в работу) снять с публикации создать вакансию закрыть/открыть вакансию выполнять действия со специалистами вакансии (описано ниже) | просмотреть выполнять действия с вакансиями (описано ниже) |
| В работе | Проект, отмеченный пользователем как “В работе” | просмотреть завершить проект создать вакансию закрыть/открыть вакансию выполнять действия со специалистами вакансии (описано ниже) перейти в чат проекта | просмотреть выполнять действия с вакансиями (описано ниже) |
| Завершен | Проект, работы над которым закончены | просмотреть | просмотреть, если был утвержден на вакансию |

Для связи проекта и специалистов предусмотрен инструмент “Вакансии”. В одном проекте может быть несколько вакансий, при этом каждая вакансия относится только к одному проекту.

Вакансия может находиться в следующих статусах:

| Статус | Описание | Доступные действия с вакансией |  |
| :---- | :---- | :---- | :---- |
|  |  | **Заказчик (создатель)** | **Специалист** |
| Открыта | Вакансия, открытая для набора | закрыть просмотреть подборку специалистов: пригласить специалиста из подборки скрыть специалиста из подборки просмотреть специалистов, которым отправлено приглашение: отозвать приглашение просмотреть откликнувшихся специалистов: принять специалиста отклонить специалиста просмотреть принятых специалистов: удалить принятого специалиста перейти в конфиденциальный чат со специалистом | просмотреть открытые вакансии: откликнуться (при совпадении специальности) перейти в конфиденциальный чат с заказчиком добавить/удалить из избранного просмотреть вакансии, в которых принимает участие: перейти в конфиденциальный чат с заказчиком добавить/удалить из избранного выйти из вакансии посмотреть вакансии, на которые пригласили: отказаться участвовать (отправить отклик) перейти в конфиденциальный чат с заказчиком добавить/удалить из избранного посмотреть вакансии, на которые отправлен отклик: отозвать отклик перейти в конфиденциальный чат с заказчиком добавить/удалить из избранного просмотреть вакансии, добавленные в избранное удалить из избранного  |
| Закрыта | Вакансия, закрытая для набора | открыть посмотреть принятых специалистов: прекратить участие принятого специалиста (для проекта в статусе “Опубликован”, “В работе”) перейти в конфиденциальный чат со специалистом | просмотреть (для проекта в статусе “Опубликован”, “В работе”) выйти из вакансии (для проекта в статусе “Опубликован”, “В работе”) посмотреть вакансии завершенных проектов (в которых принимал участие) |

### Создание проектов {#создание-проектов}

Создание проекта доступно только заказчику. При создании проекта пользователь должен указать следующую информацию:

| Данные | Обязательно / необязательно | Комментарий |
| :---- | :---- | :---- |
| Название проекта | Обязательно | Строка |
| Примерная дата старта | Обязательно |  |
| Описание | Обязательно | Для ввода описания необходимо предоставить редактор. Описание редактора приведено в разделе “[Редактор текста](#редактор-текста)”. |
| Файлы | Необязательно | Загрузка файлов. Требования к загружаемым файлам: расширения: pdf, docx, xlsx, pptx, jpg, png, txt; размер: до 200 Мб включительно (суммарно за все файлы); количество: до 20 штук. У пользователя должна быть возможность удалить загруженный файл. |
| Индустрия | Обязательно | Выбор из справочника, множественный выбор |
| Тип проекта | Обязательно | Выбор из справочника, множественный выбор  |

Система должна предоставлять возможность сохранять проекты.

Созданному проекту до публикации присваивается статус “Черновик”. 

### Редактирование проектов {#редактирование-проектов}

Редактировать проекты можно в статусах: “Черновик”.

Редактировать проект может только заказчик, создавший его.

Для редактирования доступны все поля, заполняемые при создании.

### Создание вакансий {#создание-вакансий}

Создавать вакансии для проекта может только заказчик, создавший его. 

Для создания вакансий заказчик должен указать следующую информацию:

| Данные | Обязательно / необязательно | Комментарий |
| :---- | :---- | :---- |
| Специализация | Обязательно | Строка |
| название специализации  | Обязательно | Выбор из справочника |
| уровень специализации  | Необязательно | Выбор из списка: junior middle senior lead |
| Обязанности | Обязательно | Текст, количество символов \- не более 1500\. Необходимо предоставить текстовый редактор. Подробное описание в разделе “[Редактор текста](#редактор-текста)”. |
| Языки |  |  |
| язык  | Необязательно | Выбор из справочника |
| уровень  | Необязательно, в т.ч. в случае заполнения названия языка | Выбор из справочника |
| Навыки специалиста | Обязательно | Выбор из справочника |

| Длительность проекта | Обязательно | Число. Ед.изм.: день |

Созданная вакансия получает статус “Открыта”.

Создание вакансий доступно для проектов со статусами:

- “Черновик”;  
- “Опубликованные”;  
- “В работе”.

### Редактирование вакансий {#редактирование-вакансий}

Редактировать вакансию может только заказчик, создавший ее. Редактирование вакансий возможно только в проектах со статусом “Черновик”.

Для редактирования доступны все поля, заполняемые при создании вакансии.

### Удаление проекта {#удаление-проекта}

У заказчика должна быть возможность удалять проекты. Удалять можно только проекты в статусе “Черновик”.

Удалить проект может только пользователь, создавший его.

Пользователь не видит проект после удаления.

### Публикация проекта {#публикация-проекта}

Опубликовать проект из статуса “Черновик” может только заказчик, создавший его.

К публикации допускаются проекты, в которых:

- заполнены обязательные поля;  
- есть хотя бы одна вакансия.

Опубликованный проект переходит в статус “Опубликован” и становится доступным для поиска специалистами.

### Перевод проекта в статус “В работе” {#перевод-проекта-в-статус-“в-работе”}

У заказчика должна быть возможность перевести проект из статуса “Опубликован” в статус “В работе”.

### Снятие проекта с публикации {#снятие-проекта-с-публикации}

У заказчика должна быть возможность убрать свой проект, находящийся в статусе “Опубликован”, с публикации. В таком случае проект переходит в статус “Завершен”.

Реакции системы, отправляемые на событие “Проект сняли с публикации” описаны в разделе “[События](#события)”.

### Просмотр проекта заказчиком {#просмотр-проекта-заказчиком}

У заказчика должна быть возможность просмотра созданного им проекта:

- название;  
- имя, фамилия, фотография заказчика с возможностью перехода в профиль;  
- дата старта (предполагаемого \- для проектов в статусе “Черновик”, “Опубликован”, “Завершен”, который был завершен из “Опубликован”, фактического \- для проектов в статусе “В работе”, “Завершен” из статуса “В работе”);  
- дата публикации проекта \- для проектов в статусе “Опубликован”, “В работе”;  
- дата завершения (для проектов в статусе “Завершен”);  
- описание;  
- индустрии;  
- типы;  
- файлы;  
- вакансии проекта.

### Просмотр вакансий проекта заказчиком {#просмотр-вакансий-проекта-заказчиком}

У заказчика должна быть возможность просмотреть список вакансий созданного им проекта:

- специализация:  
  - название специализации;  
  - уровень специализации;  
- навыки;  
- ставка:  
  - размер ставки;  
  - период;  
- планируемая длительность (дней).  
- описание обязанностей (текст);  
- языки:  
  - язык;  
  - уровень владения (при наличии).

### Закрытие вакансии {#закрытие-вакансии}

У заказчика должна быть возможность закрыть вакансию. Закрытая вакансия переходит в статус “Закрыта” и становится недоступна для поиска. Отправленные приглашения отменяются, отклики отклоняются. Если вакансия вновь открывается, то отклики и приглашения не возвращаются. 

### Завершение проекта {#завершение-проекта}

У заказчика должна быть возможность завершить проект, находящийся в статусе “В работе”. Проект переходит в статус “Завершен”.

Все вакансии завершенного проекта переходят в статус “Закрыта”.

Завершенный проект нельзя перевести в другой статус.

После завершения проекта система должна предлагать сделать пожертвование для сервиса. (Подробнее в разделе “[Пожертвования](#пожертвования)”). 

Список дополнительных реакций системы на событие “Проект завершен” приведен в разделе “[События](#события)”.

### Блокировка проекта {#блокировка-проекта}

У администратора должна быть возможность заблокировать проект.

Заблокированный проект становится недоступным для поиска в web-приложении. Проект переходит в статус “Заблокирован” и отображается у Заказчика во вкладке “Завершённые”.

Список дополнительных реакций системы на событие “Проект заблокировали” приведен в разделе “[События](#события)”.

### Просмотр списка проектов заказчиком {#просмотр-списка-проектов-заказчиком}

У заказчика должна быть возможность просмотра созданных им проектов. Проекты выводятся по статусам. 

- черновики;  
- опубликованные;  
- в работе;  
- завершенные (в том числе заблокированные).

Система должна информировать о количестве проектов в каждом статусе.

Для каждого проекта в списке доступна следующая информация:

- наименование проекта;  
- список индустрий (отсортированы в порядке, указанном в справочнике);  
- типы проекта (отсортированы в порядке, указанном в справочнике);  
- дата публикации проекта \- для проектов в статусе “Опубликован”;  
- дата старта (предполагаемого \- для проектов в статусе “Опубликован”, реального \- для проектов в статус “В работе”);  
- дата завершения проекта \- для проектов в статусе “Завершен”;  
- список вакансий:  
  - специализация (название и уровень);  
  - количество утвержденных специалистов;  
- количество откликов (при наличии, суммарных для всех вакансий);  
- количество новых сообщений (суммарно в личных чатах со специалистами из подборки, откликов, приглашений и утверждений на вакансии проектов \- для статуса “Опубликован”).

У заказчика должна быть возможность перейти к подробной информации о проекте.

При отсутствии проектов определенного статуса для просмотра, система должна предлагать:

- “Опубликован” \- перейти к созданию проекта;  
- “В работе” \- перейти к приглашению пользователя на платформу;  
- “Завершен” \- перейти к приглашению пользователя на платформу;  
- “Черновики” \- перейти к созданию проекта.

Заказчику должна быть доступна сортировка проектов по дате от более новых к более старым:

- опубликованные \- по дате публикации;  
- в работе \- по дате старта проекта;  
- завершенные \- по дате завершения проекта;  
- черновики \- по дате создания черновика.

## Поиск специалистов {#поиск-специалистов}

### Просмотр списка специалистов, подобранных на вакансию {#просмотр-списка-специалистов,-подобранных-на-вакансию}

Система должна предоставлять список подходящих специалистов на вакансию. Специалисты подбираются по следующим критериям:

- специализация:  
  - название специализации \- название минимум одной из специализаций пользователя точно совпадает;  
  - статус поиска проектов \- сначала отображаются специалисты со статусом “Ищу проекты”, а затем “Не ищу проекты”;  
  - мэйты \- сортировка внутри статуса поиска проектов \- отсортированы по кругу мэйта специалиста относительно заказчика \- в первую очередь \- первый круг, после \- второй, после \- остальные пользователи.

Система должна предоставлять заказчику возможность просмотра специалистов, подобранных на его вакансию. О каждом специалисте должна быть доступна информация, с возможностью перехода к подробной информации о специалисте:

- фотография специалиста;  
- имя специалиста;  
- круг мэйта, в котором состоит специалист, относительно заказчика.

Если специалист выступает в web-приложении в двух ролях \- то при переходе к подробной информации о специалисте отображается информация профиля “специалист”.

У заказчика должны быть следующие возможности для каждого специалиста, подобранного на вакансию:

- пригласить специалиста (Подробно о приглашении в разделе “[Приглашение специалистов](#приглашение-специалистов-на-вакансию)”);  
- скрыть специалиста из рекомендаций.

Если заказчик в приложении выступает в двух ролях, то ему не отображается его профиль специалиста в списке подобранных специалистов.

**Скрыть специалиста из рекомендаций**

Заказчик может скрыть специалиста из рекомендаций. Скрытый специалист не отображается в подборке по этой вакансии, но сможет появиться в других подборках заказчика.

### Просмотр вакансий специалистом {#просмотр-вакансий-специалистом}

У специалиста должна быть возможность просмотра открытых вакансий. 

Просматривать вакансии, созданные другими пользователями можно только в роли “Специалист”. 

По умолчанию вакансии сортируются по следующим правилам:

- специализация:  
  - специализация \- в первую очередь выводятся вакансии, специализация которых полностью совпадает с одной из специализаций специалиста;  
- круг мэйта заказчика \- отсортированы по кругу мэйта в котором находится заказчик относительно специалиста \- в первую очередь выводятся мэйты первого уровня, далее \- второго, после \- третьего.

Если у специалиста не опубликован профиль специалиста, то выводятся все вакансии, отсортированные по дате публикации проекта по убыванию.

В списке вакансий о каждой вакансии выводится следующая информация:

- данные заказчика с возможностью перехода в профиль заказчика:  
  - фотография заказчика;  
  - имя заказчика;  
  - фамилия заказчика;  
  - круг мэйта, в котором находится заказчик относительно специалиста;  
- данные проекта с возможностью перехода к проекту:  
  - название проекта;  
  - типы проекта (в порядке, заданном в справочнике);  
  - индустрии (в порядке, заданном в справочнике);  
  - дата публикации проекта;  
  - дата старта (предполагаемого \- для статуса “Опубликован”, фактического \- для статуса “В работе”);  
- информация о подобранной вакансии:  
  - уровень вакансии (при наличии);  
  - специализация вакансии;  
  - ставка вакансии;  
  - количество остальных вакансий проекта.

При переходе к подробной информации о вакансии из списка (подробнее в разделе “[Просмотр проекта специалистом](#просмотр-проекта-специалистом)”) первой отображается вакансия, из которой был выполнен переход.

Если пользователь выступает в приложении в двух ролях, то в роли специалиста ему недоступны для просмотра вакансии проекта, которые создал он в роли заказчика.

**Подборка вакансий**

В подборке вакансий пользователь может просмотреть:

- вакансии в статусе “Открыта” проектов в статусе “Опубликован”, “В работе”. 

**Избранное**

В избранном пользователь может просмотреть вакансии, которые он добавил в избранное:

- вакансии в статусе “Открыта” проектов в статусах “В работе”, “Опубликован”;  
- вакансии в статусе “Закрыта” проектов в статусах “В работе”, “Опубликован”;  
- вакансии проектов в статусе “Завершено”, только если пользователь принимал в них участие и не был удален / не выходил из вакансии.

**Список вакансий, в которых принимает участие**

В списке пользователь может просмотреть:

- вакансии, независимо от их статуса, на которые утвержден пользователь, проектов в статусах “В работе”, “Опубликован”.

**Список вакансий, на которые отправлены отклики**

В списке пользователь может просмотреть:

- вакансии в статусе “Открыта”, на которые пользователь отправил отклик, проектов в статусах “В работе”, “Опубликован”.

**Список вакансий завершенных проектов**

В списке пользователь может просмотреть:

- вакансии проектов в статусе “Завершено”, в которых пользователь принимал участие и не был удален / не выходил из вакансии.

### Добавление вакансии в избранное {#добавление-вакансии-в-избранное}

У специалиста должна быть возможность отметить вакансию как “Избранная”.

### Удаление вакансии из избранного {#удаление-вакансии-из-избранного}

У специалиста должна быть возможность удалить вакансию из избранных.

### Фильтр вакансий {#фильтр-вакансий}

Специалист должен иметь возможность отфильтровать доступные для поиска вакансии.

Система предоставляет список вакансий, подходящих под условия фильтра:

- круг мэйта заказчика относительно специалиста \- в первую очередь мой круг, потом 2-й и затем 3-й;  
- специализация \- в первую очередь \-  полное совпадение с одной из специализаций специалиста;  
- уровень специализации \- полное совпадение;  
- ставка от и до \- значения попадающие в указанный диапазон (включительно);  
- дней на вакансии (“от” и “до” включительно);  
- тип проекта вакансии \- если хотя бы один из указанных исполнителем совпадает с указанным в проекте;  
- навыки \- если хотя бы один указанных исполнителем навык совпадает;  
- индустрия проекта;  
- дата старта проекта \- проекты от указанной даты (включительно).

У специалиста должна быть возможность сбросить фильтры к начальным значениям.

### Сортировка вакансий {#сортировка-вакансий}

У специалиста должна быть возможность отсортировать вакансии:

- сначала новые;  
- по убыванию цены;  
- по возрастанию цены.

При сортировке “Сначала новые” сначала отображаются более свежие вакансии, затем более старые:

- в подборке \- по дате создания вакансии, учитывая правила сортировки по совпадению специализации и круга мэйта заказчика (описаны в [Просмотр вакансий специалистом](#просмотр-вакансий-специалистом));  
- в избранном \- по дате добавления в избранное;  
- участие \- по дате утверждения заказчиком специалиста;  
- приглашения \- по дате создания приглашения;  
- мои отклики \- по дате создания отклика;  
- завершенные \- по дате завершения проекта.

### Просмотр проекта специалистом {#просмотр-проекта-специалистом}

Система должна предоставлять специалисту возможность просмотра информации о проекте.

При переходе к проекту должна быть доступна информация:

- данные проекта:  
  - дата публикации проекта (для проектов в статусе “Опубликован”, “В работе”);  
  - дата старта (предполагаемого \- для проектов в статусе “Опубликован”, фактического \- для проектов в статусе “В работе”, “Завершен”);  
  - дата завершения (для проектов в статусе “Завершен”);  
  - название проекта;  
  - индустрии (в порядке, заданном в справочнике);  
  - типы проекта (в порядке, заданном в справочнике);  
  - описание проекта;  
  - файлы проекта;  
- данные заказчика с возможностью перехода в профиль заказчика:  
  - фотография заказчика;  
  - имя и фамилия заказчика;  
  - круг мэйта, в котором состоит заказчик, относительно специалиста;  
- вакансии проекта.

**Список вакансий проекта**

О каждой вакансии в проекте выводится следующая информация:

- специализации:  
  - название специализации;  
  - уровень (при наличии);  
- навыки (в порядке, заданном в справочнике);  
- ставка:  
  - размер ставки;  
  - период;  
- планируемая длительность;  
- текст с описанием обязанностей;  
- языки:  
  - язык;  
  - уровень;  
- специалисты, утвержденные на вакансию \- информация аналогична информации в разделе “[Просмотр списка специалистов, подобранных на вакансию](#просмотр-списка-специалистов,-подобранных-на-вакансию)”;  
- информация о том, что вакансия закрыта (при наличии).

## Приглашения на вакансию {#приглашения-на-вакансию}

У заказчика есть возможность:

- пригласить специалистов на вакансию;  
- просмотреть список специалистов, которых он пригласил на вакансию;  
- удалить приглашение на вакансию.

У специалиста должна быть возможность:

- посмотреть список проектов, на вакансию в которые его пригласили;  
- отказаться от приглашения на вакансию;  
- принять приглашение на вакансию.

### Приглашение специалистов на вакансию {#приглашение-специалистов-на-вакансию}

Заказчик может приглашать специалистов, если их специализация совпадает с указанной в вакансии.

Система должна предоставлять возможность оставлять необязательный текстовый комментарий при отправке приглашения. Длина комментария должна быть до 256 символов включительно. Добавленный комментарий отправляется в личные сообщения специалисту.

Дополнительные реакции системы на событие “Пригласили на вакансию” перечислены в разделе “[События](#события)”.

### Просмотр приглашенных {#просмотр-приглашенных}

У заказчика должна быть возможность просмотреть список приглашенных им на вакансию специалистов.

О каждом специалисте выводится информация, аналогичная информации, описанной в разделе “[Просмотр списка специалистов, подобранных на вакансию](#просмотр-списка-специалистов,-подобранных-на-вакансию)”. 

Для проектов в статусе “Завершен” список приглашенных не отображается. 

### Удаление приглашения {#удаление-приглашения}

У заказчика должна быть возможность удалить отправленное им приглашение. Приглашение можно удалить, пока специалист не отказался или не принял приглашение.

Дополнительные реакции на событие “Приглашение отменили” перечислены в разделе “[События](#события)”.

### Просмотр приглашений специалистом {#просмотр-приглашений-специалистом}

У специалиста должна быть возможность просмотра вакансий и проектов вакансий, в которые его пригласили. 

О каждом проекте должна быть доступна информация, аналогичная информации разделов “[Просмотр вакансий специалистом](#просмотр-вакансий-специалистом)”, “[Просмотр проекта специалистом](#просмотр-проекта-специалистом)”. 

Система должна предоставлять информацию о количестве приглашений.

Если проект был закрыт, он перестает отображаться в списках приглашений.

### Отказ от приглашения {#отказ-от-приглашения}

У специалиста должна быть возможность отказаться от вакансии, на которую его пригласили.

После отказа специалист не отображается в подборках специалистов на вакансию.

Дополнительные реакции системы на “Специалист отказался от приглашения” перечислены в разделе “[События](#события)”.

Принятие приглашения

У специалиста должна быть возможность принять приглашение на вакансию, на которую его пригласили.

**После принятия приглашения:**

- проект с вакансией переходит в список откликов у специалиста;  
- специалист переходит в список откликнувшихся специалистов на вакансию у заказчика.

## Отклик на вакансию {#отклик-на-вакансию}

### Оставить отклик на вакансию {#оставить-отклик-на-вакансию}

У специалиста должна быть возможность оставить отклик на вакансию.

Специалист может оставить отклик только на вакансию, соответствующую одной из его специализаций. При отсутствии подходящей специализации система сообщит о необходимости её добавить.

Из списка откликов специалист может перейти в конфедициальный чат с заказчиком, на чей проект он откликнулся.

Дополнительные реакции системы на событие “Откликнулись на вакансию” перечислены в разделе “[События](#события)”.

### Просмотр своих откликов {#просмотр-своих-откликов}

Система должна предоставлять специалисту возможность просмотра вакансий, на которые он откликнулся.

О каждой вакансии и проекте вакансии должна быть доступна информация, аналогичная информации разделов “[Просмотр вакансий специалистом](#просмотр-вакансий-специалистом)”, “[Просмотр проекта специалистом](#просмотр-проекта-специалистом)”. 

При закрытии проекта он перестает отображаться в списках откликов специалистов.

### Отмена отклика специалистом {#отмена-отклика-специалистом}

Система должна предоставлять возможность отозвать свой отклик. Отозвать отклик можно только для вакансий со статусом “Открыта”. 

Дополнительные реакции системы на “Специалист отозвал отклик” перечислены в разделе “[События](#события)”.

### Просмотр откликнувшихся специалистов {#просмотр-откликнувшихся-специалистов}

Система должна предоставлять заказчику список откликнувшихся на размещенную им вакансию.   
О каждом специалисте доступна информация, аналогичная разделу “[Просмотр списка специалистов, подобранных на вакансию](#просмотр-списка-специалистов,-подобранных-на-вакансию)”.

Для каждого специалиста доступны действия:

- принять \- утвердить на вакансию;  
- отклонить \- отклонение отклика специалиста.

Система должна информировать о количестве откликов.

После закрытия проекта список откликнувшихся пользователей на вакансию не отображается.

### Отклонение откликнувшегося специалиста {#отклонение-откликнувшегося-специалиста}

У заказчика должна быть возможность отклонить специалиста, оставившего отклик на вакансию. 

Отклоненный специалист может откликнутся еще раз.

Система должна предоставлять возможность добавлять необязательный комментарий к отклонению отклика. Длина комментария должна быть до 256 символов включительно.

Дополнительные реакции системы на событие “Заказчик отклонил отклик” перечислены в разделе “[События](#события)”.

### Удаление утвержденного специалиста {#удаление-утвержденного-специалиста}

Дополнительные реакции системы на событие “Заказчик отменил участие в проекте” перечислены в разделе “[События](#события)”.

Список дополнительных реакция системы на событие “Специалист ушел с вакансии” прописано в разделе “[События](#события)”.

## Чат {#чат}

В системе должен быть предусмотрен механизм обмена сообщениями между участниками \- механизм чатов.

Если пользователь находится в роли заказчика, то ему доступны только чаты, доступные для заказчика. А если в роли специалиста, то \- только чаты для специалиста. 

Система должна предоставлять возможность:

- начало чата с пользователем;  
- просмотра чатов пользователя;  
- просмотра сообщений чата, в котором состоит пользователь;  
- отправка сообщений в чат, в котором состоит пользователь.

Система должна уведомлять о новых сообщениях в чате.

Пользователь может видеть только те чаты, в которых он состоит или состоял.

Чаты пользователей должны быть отсортированы по дате появления в них новых сообщений в порядке убывания.

**Просмотр сообщений чата**

О каждом сообщении должна быть доступна следующая информация:

- автор сообщения  
  - фамилия, имя автора;  
  - фото профиля автора;  
- дата и время сообщения;  
- текст сообщения;  
- статус сообщения: прочитано / не прочитано.

При просмотре сообщений чата система должна показывать сообщения, начиная с непрочитанных пользователем.

**Типы сообщений**

Система должна поддерживать только текстовые сообщения.

### Виды чатов {#виды-чатов}

Предусмотрены следующие виды чатов:

- конфиденциальные чаты;  
- чаты проекта;  
- чат с поддержкой (готовое решение);  
- чат уведомлений.

**Конфиденциальные чаты**

Система должна предоставлять возможность пользователю создавать конфиденциальные чаты с любым из участников системы.

**Чаты проекта**

При переводе проекта в статус “В работе” система должна создавать чат для него. Название чата должно совпадать с названием проекта.

Система должна добавлять создателя проекта в чат сразу при создании.

Система должна добавлять в чат всех пользователей, утвержденных на вакансии проекта.

Если специалист был утвержден на вакансию проекта в статусе “В работе”, то ему недоступна история сообщений до его добавления в чат.

Пользователь не может сам выйти из чата проекта.

Система должна удалять из чата пользователей, удаленных из проекта. Удаленные пользователи могут видеть только переписку до момента выхода из чата.

Система должна закрывать чаты при закрытии проекта. Закрытые чаты остаются доступны для просмотра, но недоступны для отправки новых сообщений.

**Чат с поддержкой**

*Ранее было подключено готовое решение – Jivo (JivoSite), но оно было отключено по запросу заказчика.*

**Чат уведомлений**

При регистрации для каждого пользователя создается чат, в который:

- отправляются все уведомления системы;  
- отправляются уведомления, написанные администратором в административной панели.

При получении нового сообщения от администратора в чате уведомлений система должна уведомлять пользователя.

Пользователь не может покинуть чат уведомлений. 

У пользователя не должно быть возможности отправлять сообщения в чат уведомлений.

Список возможных уведомлений приведен в разделе “[События](#события)”.

Удаленные пользователем проекты не отображаются в административной панели и не отображаются в интерфейсе web-приложения.

**Типы проектов**

- просмотр типов проектов в списке и детально;

- дата последнего входа в web-приложение.

При детальном просмотре пользователя заказчика должно быть доступно:

- о себе.

**Профили фрилансеров**

Информация, доступная для просмотра в списках:

### Блокировка пользователя

У администратора должна быть возможность заблокировать пользователя.

Если у пользователя есть проекты, отличные от статуса “Завершен” \- они переводятся в статус “Заблокирован” и отображаются во вкладке “Завершен”. 

Если пользователь специалист и участвует в проектах \- его отклики становятся недоступны для просмотра в web-приложении. Пользователь удаляется с вакансий навсегда.

Заблокированного пользователя перенаправляет на экран авторизации. При попытке войти отображается ошибка: “Ваш профиль заблокирован”.

Профиль заблокированного пользователя становится недоступным для поиска и просмотра. 

Личные чаты с удаленным пользователем остаются доступными только для просмотра. Сообщения из чатов проектов, в которых состоял пользователь, остаются, их автор отмечается как “Заблокированный”.

Дополнительная реакции на событие “Заказчика заблокировали” описано в разделе “[События](#события)”.

Дополнительные реакции системы на событие “Специалиста заблокировали” приведено в разделе “[События](#события)”.

### Настройка справочников {#настройка-справочников}

Система должна предоставлять возможность управлять следующими видами справочников:

- индустрии;  
- навыки;  
- опыт работы;  
- специализации;  
- языки:  
  - названия языка;  
  - уровни владения языком.

О каждом справочнике для просмотра доступна следующая информация:

- название справочника;  
- история изменений.

Для каждого справочника доступны следующие действия:

- добавить позицию в справочник;  
- удалить позицию из справочника;  
- редактировать позицию в справочнике.

### Чат уведомлений

Система должна предоставлять администратору возможность отправлять сообщения о новостях и акциях всем пользователям в их чат уведомлений.

В административной панели в разделе "Сообщения" должен быть доступен функционал создания и отправки сообщения в чаты уведомлений пользователей.

Администратор имеет возможность:

- написать текстовое сообщение (без вложений);  
- отправить сообщение всем пользователям.

Отправленные администратором сообщения должны отображаться в чатах уведомлений пользователей так же, как системные уведомления.

### Отчеты {#отчеты}

В системе должна быть возможность формирования отчетов с возможностью выгружать отчеты в формате csv.

Отчеты предполагается строить за выбранный период времени.

Список выгружаемых отчетов:

- скорость отклика фрилансеров на проекты;  
- скорость запуска проекта (с момента публикации до старта работ);  
- доля отмененных проектов до и после запуска;  
- доля запущенных проектов с рекомендованными специалистами (2й круг / 3й круг);  
- доля случаев, когда после завершения проекта с рекомендованными специалистами (2й круг / 3й круг), пользователи настраивают связь;  
- доля самых активных пользователей (про метрику возврата в сервис и создание/отклик на новые проекты);   
- среднее кол-во отправленных приглашений в сервис на 1 пользователя;  
- среднее кол-во отправленных запросов на добавление в 1й круг;  
- доля проектов с командой \>1 специалистов (не считая заказчика);  
- доля завершенных проектов с донатом;  
- средний чек доната.

Список отчетов, не вошедших в mvp:

- доля выполненных проектов, закрытых в срок \- в данной версии системы нет понятия срока проекта. Есть понятие: срок вакансии, но нет возможности   
  отследить, когда работы на ней будут выполнены. Вакансия закрывается вместе с проектом;  
- доля негативных отзывов по специалисту \- в данной версии нет возможности оставлять отзывы специалистам;  
- индекс удовлетворенности специалистом \- в данной версии нет возможности измерения индекса удовлетворенности, откликов, оценок;  
- доля проектов со спорными случаями (эскалирование на арбитраж) \- в текущей версии системы нет инструмента для отслеживания спорных случаев.
